# 2026-05-11 File Management Plan and Context

## Goal

Add robust file management to MindMapHelper with:

- Native save/load in our own format (JSON).
- Import support from other mind-map applications through adapters.
- Single-document runtime model (one app instance handles one open file).

## Current Implementation Context

### Runtime and Ownership

- `src/main.cpp` ignores CLI arguments and calls `mind_map::app::RunApp()`.
- `src/app/AppMain.h` and `src/app/AppMain.cpp` expose `RunApp()` with no document/session input.
- UI state is process-lifetime static in `src/ui/MindMapUi.cpp` (`static UiState state;` in `RenderMainUi()`).
- `UiState` owns:
  - `MindMapCanvasView canvas_`
  - `pan_px_`
  - `zoom_`
  - `show_status_bar_`

### Editing and Data Flow

- Commands are local to `src/ui/MindMapUi.cpp` via `UiCommandId` + `UiCommandDispatcher`.
- Existing File menu currently exposes reset actions, not file I/O.
- `MindMapCanvasView` (in `src/ui/MindMapCanvasView.h/.cpp`) owns interactive map state:
  - Node positions
  - Drag/select interaction state
  - Per-child branch style state

### Topology and Model Constraints

- The graph currently behaves as a sample/static topology setup.
- Topology single source of truth: `src/core/mindmap/SampleMindMapTopology.h/.cpp`; `SampleMindMapGraph.h` re-exports via `using` and now only contains `HitTestSampleMap` + `InitialSampleMapPositions` (both deleted in step 2).
- Stable geometry (rounded-rect attachment, bezier arms, etc.) lives in `src/ui/canvas/NodeGeometry.h` (`mind_map::canvas` namespace); `BranchEdgeAttachments.h` consumes it directly with no dependency on demo code.
- `UiState` is defined in `src/ui/UiState.h`; owned by `AppMain::RunApp()` and passed into `RenderMainUi(UiState&)` each frame — no static locals.
- No persistence infrastructure currently exists:
  - No document repository
  - No serializer/deserializer
  - No import pipeline
  - No migration/versioning path

### Architectural Constraints (AGENTS.md alignment)

- `core` must remain platform-neutral and must not depend on ImGui/UI/platform APIs.
- `ui` can depend on `core`.
- `platform` owns window creation and GL context; file dialog is **not** platform-owned (see Dependencies below).
- `app` should orchestrate lifecycle/session behavior.

### Dependencies (decided)

| Dependency | How added | Used by | Notes |
|---|---|---|---|
| `nlohmann/json` | `FetchContent_Declare` + `FetchContent_MakeAvailable` | `mindmap_helper_core` | Header-only; provides CMake target `nlohmann_json::nlohmann_json` |
| `ImGuiFileDialog` | `FetchContent_Declare` + `FetchContent_MakeAvailable` | `mindmap_helper_ui` | Source added to `BM_UI_SOURCES` manually, same pattern as `IMGUI_SOURCES`; ImGui widget — renders inside the frame loop, lives in `ui` layer not `platform` |

All dependencies use `FetchContent` — consistent with glfw and imgui. No git submodules, no `third_party/` directory.

**Node/edge identifiers:** UUID v4 strings (`xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx`), generated in-process using `<random>`. No extra library required — a small static helper (~20 lines) in `src/core/mindmap/` is sufficient.

## Why Single-Document Mode Is a Good Fit

The "mono file" model (one running instance per file) simplifies v1:

- No multi-tab or document-switching logic.
- No active-document routing complexity.
- Fewer unsaved-change edge cases.
- Clear lifecycle: `New`, `Open`, `Save`, `Save As`, `Import`, `Close`.

This reduces surface area, but still benefits from a proper document/session architecture.

## Design Plan (Patterns + Responsibilities)

### 1) Canonical Document Model (Core)

Create a neutral domain model in `src/core`:

- `MindMapDocument`
- Stable node/edge identifiers: **UUID v4 strings** generated by `UuidGenerator` helper in `src/core/mindmap/` using `<random>` — no external library
- Layout/style data represented explicitly
- Format/version metadata

Pattern: **Document Model**.

### 2) Native Persistence Abstraction

Add repository interface + JSON implementation:

- `IDocumentRepository` (`Load`, `Save`)
- `JsonNativeDocumentRepository`
- `DocumentMigration` utilities for version upgrades

Pattern: **Repository**.

### 3) Import Pipeline for External Formats

Introduce adapter-based import service:

- `IImportAdapter` per external format/application
- `ImportService` selects adapter (extension/content sniffing)
- Importers output canonical `MindMapDocument`

Pattern: **Adapter Strategy**.

### 4) Session/Application Service

Add app-level session coordinator in **`src/app/`**:

- `DocumentSessionService`
- Owns `current_path_`, `dirty_`, and operation state
- `dirty_` is set to `true` by every document-mutating operation (node drag, style change, etc.) and cleared on successful Save/SaveAs/New/Open
- Provides use cases:
  - `New` — confirmation guard if dirty, then full canvas replace with empty document
  - `Open` — confirmation guard if dirty, then full canvas replace with loaded document
  - `Save` / `SaveAs`
  - `Import`
  - `RequestClose()` — flag checked by `RenderMainUi()` to show confirmation modal; window-close guard intercepts GLFW close path
- Depends on `IDocumentRepository` for all Load/Save I/O; never calls file dialogs directly
- `RenderMainUi()` must be updated to accept `DocumentSessionService&` so the File menu and close-guard modal can call it

Pattern: **Application Service / Use-Case Layer**.

### 5) Document / UI State Split

Separate `MindMapCanvasView` state by one criterion: **would losing this state lose user work?**

| Field | Category | On load |
|---|---|---|
| Node positions | document state | replaced by loaded values |
| Branch styles per edge | document state | replaced by loaded values |
| `pan_px_`, `zoom_` | viewport metadata | replaced (stored in JSON `viewport` section) |
| `selected_child_for_edge_` | interaction state | reset to "none" |
| drag state | interaction state | reset to "not dragging" |

**Code structure:** extract a plain-data struct (or use `MindMapDocument` directly) that the canvas can be initialised from on load and serialised from on save. Keep a separate `MindMapInteractionState` for ephemeral concerns. Only document state + viewport metadata participate in serialisation.

**`Reset()` / load semantics:** `Open` and `New` both perform a **full canvas replace** — no partial merge. The sequence is:
1. If `dirty_` is true, show "Unsaved changes — save before continuing?" (Save / Discard / Cancel). Abort on Cancel.
2. Call `canvas.LoadFrom(document_state)`: replaces all document-state fields and resets all interaction fields to zero/default.

This means topology differences between the current and incoming document are irrelevant — the old state is gone entirely. `File > New` follows this same path with a minimal `MindMapDocument` as input — one root node with a generated UUID and label "New Mind Map".

**Demo initialisation persistence:** Until node-creation UI exists, the app must have something to display at startup. `BuildSampleDocument()` in `src/ui/demos/` converts `SampleMindMapTopology` + `InitialSampleMapPositions()` into a `MindMapDocument` (including stable per-edge style defaults and UUIDs). `MindMapCanvasView()` calls `LoadFrom(BuildSampleDocument())`. `Reset()` continues to reset only positions and interaction state (not styles), so it remains "Reset Layout" semantically. `SampleMindMapTopology.h/.cpp`, `InitialSampleMapPositions()`, and `BuildSampleDocument()` are deleted together when node-creation UI lands — at that point startup goes through `File > New`.

Pattern: **Presentation Model** — pure data model, view adds interaction on top.

### 6) DTO Mapping Boundary

Keep JSON DTOs separate from in-memory domain structures and map explicitly.

Pattern: **Mapper Boundary** for compatibility and migration safety.

### 7) File Dialog (UI Layer)

File open/save dialogs use **ImGuiFileDialog** (added via FetchContent), rendered as an ImGui widget inside the normal frame loop. This means:

- No per-platform dialog code in `platform/`.
- Dialog state (open/pending/result) lives in `UiState` or a dedicated `FileDialogUiState`.
- `DocumentSessionService` receives the resolved path from the UI after the dialog closes — it never calls the dialog directly.
- Usage pattern per frame: `OpenDialog(key, …)` on trigger → `Display(key)` every frame → `GetFilePathName()` on close guard.

## Proposed Native JSON Envelope (v1)

Native file extension: **`.mmh`** (used in ImGuiFileDialog filters and as the default save suffix).

Use a strict versioned top-level shape:

- `format`: `"mindmaphelper"`
- `version`: integer
- `document`: metadata (title/timestamps if needed)
- `nodes`: stable IDs + node content
- `edges`: relationships by IDs
- `layout`: per-node coordinates
- `styles`: per-edge style data
- `viewport`: `pan_x`, `pan_y`, `zoom` (courtesy — restores the user's last view on open)

## Quality and Design Principles Check

### YAGNI

- Keep v1 single-document and defer multi-document/tab orchestration.

### SOLID

- Separate UI rendering, session orchestration, persistence, and import concerns.
- Depend on interfaces (`IDocumentRepository`, `IImportAdapter`) at orchestration boundaries.

### KISS

- One canonical document model.
- One native JSON format.
- Import = convert to canonical model, then normal save/load flow.

### DRY

- Avoid duplicating topology/data definitions across UI and core.
- Establish one source of truth for persisted structure.

## Risks to Address Early

- Existing sample/static topology approach is not yet a full dynamic document model.
- Topology duplication resolved; geometry promoted: stable algorithms live in `src/ui/canvas/NodeGeometry.h` (names stripped of `SampleMap` prefix, namespace `mind_map::canvas`). `SampleMindMapGraph.h` now contains only `HitTestSampleMap` + `InitialSampleMapPositions`; both survive until node-creation UI lands. `BuildSampleDocument()` (added in step 2) consumes them internally and is the single startup-document factory. All three are deleted together when node creation makes a startup document unnecessary.
- `Reset()` behavior resolved: `Open` and `New` both perform a full canvas replace after a dirty-check confirmation. See Design Plan §5.
- File dialog lives in `ui` (ImGuiFileDialog widget), not `platform` — old assumption corrected.
- Unsaved-changes close guard: handled inline in `AppMain.cpp` after `glfwPollEvents()` — if `glfwWindowShouldClose` is set and `session.IsDirty()`, call `glfwSetWindowShouldClose(window, 0)` to veto and `session.RequestClose()` to set a flag. `RenderMainUi()` checks the flag and opens the ImGui confirmation modal. On Discard/Save the modal calls `glfwSetWindowShouldClose(window, 1)`; on Cancel it clears the flag. No callbacks, no cross-layer wiring.
- CLI path support (step 6) is a contained 3-file change (`main.cpp`, `AppMain.h`, `AppMain.cpp`); platform bootstrap files are unaffected.
- Import format for the first adapter (step 7) is not yet chosen. Decision affects whether an XML parser or zip library is also needed.
